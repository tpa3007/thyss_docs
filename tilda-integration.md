## 1. Обзор интеграции

### 1.1. Что такое интеграция THYSS × Tilda

Интеграция позволяет принимать крипто-платежи (USDT, TON-сеть) прямо через формы и корзину Tilda — без написания кода. Покупатель оформляет заказ привычным способом, а THYSS автоматически создаёт страницу оплаты и конвертирует сумму из рублей в USDT по актуальному курсу.

> **Не требует:** сервера, программиста, технических знаний.  
> **Время настройки:** 15–20 минут.

---

### 1.2. Схема работы

```
┌─────────────────────────────────────────────────────────────┐
│                      ПОКУПАТЕЛЬ                             │
└──────────────────────────┬──────────────────────────────────┘
                           │ нажимает «Оплатить»
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                        TILDA                                │
│  Собирает данные формы → POST → api.thyss.xyz               │
└──────────────────────────┬──────────────────────────────────┘
                           │ данные заказа + подпись
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                        THYSS                                │
│  Проверяет подпись → создаёт платёжную сессию               │
│  Конвертирует RUB → USDT по текущему курсу (+1% буфер)      │
└──────────────────────────┬──────────────────────────────────┘
                           │ redirect → pay.thyss.xyz
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                   СТРАНИЦА ОПЛАТЫ HPP                       │
│  QR-код · TON-адрес · таймер 30 минут                      │
└──────────────────────────┬──────────────────────────────────┘
                           │ покупатель платит через Tonkeeper
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                   АВТОПОДТВЕРЖДЕНИЕ                         │
│  THYSS обнаруживает платёж (1–2 мин) → статус Success       │
│  Webhook на ваш сервер (если настроен)                      │
└─────────────────────────────────────────────────────────────┘
```

---

### 1.3. Требования перед началом

Перед началом настройки убедитесь, что выполнены все условия:

| # | Требование | Где проверить |
|---|---|---|
| 1 | Аккаунт THYSS зарегистрирован | dashboard.thyss.xyz |
| 2 | TON-кошелёк подключён к аккаунту | THYSS → Настройки → Кошелёк |
| 3 | Сайт на Tilda с блоком корзины или формы оплаты | Редактор Tilda |
| 4 | Public Key получен | THYSS → Настройки → Интеграция → Tilda |
| 5 | Secret Key получен | THYSS → Настройки → Разработчикам |

## 2. Получение ключей из THYSS

### 2.1. Public Key и Endpoint URL

Откройте: **THYSS Дашборд → Настройки → Интеграция → вкладка «Tilda»**

Вы увидите готовый Endpoint URL вида:

```
https://api.thyss.xyz/integrations/tilda?pk=pk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

Скопируйте его кнопкой «Копировать» — он понадобится на шаге 3.3.

> Public Key (часть после `?pk=`) — публичный идентификатор вашего аккаунта. Его безопасно вставлять в URL.

---

### 2.2. Secret Key

Откройте: **THYSS Дашборд → Настройки → Разработчикам**

Найдите поле **Secret Key**:

```
sk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

> **Важно:** Secret Key — приватный. Никому не передавайте его и не публикуйте в открытом доступе. Если подозреваете утечку — ротируйте ключ через кнопку «Сгенерировать новый».

## 3. Настройка платёжной системы в Tilda

### 3.1. Создание платёжной системы

1. Войдите в **Tilda** и откройте нужный проект
2. Перейдите в раздел **«Оплата»** проекта
3. Нажмите **«Добавить платёжную систему»**
4. Выберите **«Другие»** → **«Универсальная платёжная система»**
5. Выберите **«Новая платёжная система (для разработчиков)»**

Откроется страница конфигурации. Заполняйте поля строго по инструкции ниже.

---

### 3.2. Основные настройки

Это первый экран — базовые поля платёжной системы.

#### Логин

**Значение:** ваш Public Key из раздела 2.1

```
pk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

> Tilda требует заполнить это поле. Вставьте Public Key — скопируйте из THYSS дашборда.

---

#### Секрет для подписи заказа

**Значение:** ваш Secret Key из раздела 2.2

```
sk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

> Tilda использует этот ключ для вычисления цифровой подписи каждого заказа.

---

#### URL для уведомлений

**Значение:** *оставить пустым*

> THYSS самостоятельно отслеживает платежи — этот адрес не используется.

---

#### Валюта

**Значение:** валюта вашего магазина

| Если магазин в... | Укажите |
|---|---|
| Рублях | `RUB` |
| Долларах | `USD` |
| USDT напрямую | `USDT` |

> При `RUB` THYSS автоматически конвертирует сумму в USDT по курсу на момент оплаты.

---

#### Страна и Язык виджета оплаты

**Значение:** *не трогайте* — THYSS не использует эти поля.

---

### 3.3. API URL

Раскройте раздел **«Расширенные настройки»** → подраздел **«URL»**.

#### API URL

**Значение:** скопированный Endpoint URL из раздела 2.1

```
https://api.thyss.xyz/integrations/tilda?pk=pk_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

> ⚠️ Убедитесь, что в URL подставлен **ваш** `pk_live_...`. URL без параметра `?pk=` работать не будет.

---

### 3.4. Маппинг полей

Раздел **«Поля»** → **«Список соответствия полей»**.

Здесь указываются имена полей, под которыми Tilda отправляет данные в POST-запросе. THYSS ожидает конкретные имена — заполняйте точно по таблице.

> В каждой строке есть колонка **«Поле»** — туда и вписывайте значения из правого столбца таблицы.

| Строка в Tilda | Что вписать в «Поле» | Обязательно |
|---|---|---|
| Логин | `login` | ✅ |
| Валюта | `currency` | ✅ |
| **Номер заказа** | `orderid` | ✅ |
| **Сумма заказа** | `payment` | ✅ |
| **Подпись** | `secret` | ✅ |
| Email покупателя | `Email` | Рекомендуется |
| Телефон покупателя | `Phone` | Рекомендуется |
| Имя покупателя | `Name` | Рекомендуется |
| Товары | `products` | При наличии корзины |
| Язык | *(пусто)* | — |
| Страна | *(пусто)* | — |
| URL для уведомлений | *(пусто)* | — |
| Описание заказа | *(пусто)* | — |
| Время заказа | *(пусто)* | — |

> **Примечание:** поля Email, Phone, Name не влияют на проведение платежа, но обогащают карточку клиента в THYSS дашборде.

---

### 3.5. Маппинг товаров

Если вы используете блок корзины с товарами, заполните вложенный маппинг внутри строки «Товары»:

| Строка | Что вписать | Обязательно |
|---|---|---|
| Название | `name` | ✅ |
| Количество | `quantity` | ✅ |
| Цена | `price` | ✅ |
| Порядковый номер | *(пусто)* | — |
| Полная стоимость | *(пусто)* | — |
| Фото | *(пусто)* | — |

---

### 3.6. Настройка подписи заказа

Раздел **«Подпись заказа»** — критически важный блок. Именно здесь настраивается криптографическая защита от подделки заказов.

> ⚠️ **Ошибки в этом разделе — наиболее частая причина ошибки «Invalid signature».** Заполняйте строго по инструкции.

---

#### Правила подписи

**Значение:** `Все поля`

---

#### Исключить поля

**Значение:**

```
secret
```

> Поле `secret` содержит саму подпись — его нельзя включать в вычисление хэша.

---

#### Порядок сортировки полей

**Значение:** `В алфавитном порядке, с начала`

> THYSS сортирует поля в том же порядке перед вычислением хэша.

---

#### Разделитель

**Значение:** *оставить пустым*

> Значения полей конкатенируются без разделителя.

---

#### Добавлять секрет в подпись

**Значение:** `Последним элементом`

> THYSS ожидает Secret Key в конце строки перед хэшированием: `[значения полей по алфавиту] + [Secret Key]`.

---

#### Алгоритм шифрования подписи

**Значение:** `SHA-256`

> ⚠️ По умолчанию в Tilda стоит **MD5** — его **обязательно** нужно изменить на **SHA-256**. Это самая распространённая ошибка при настройке.

---

#### Чекбоксы алгоритма

| Опция | Значение |
|---|---|
| Использовать секрет в качестве ключа алгоритма (HMAC) | ❌ Выключено |
| Преобразовать к верхнему регистру | ❌ Выключено |
| Преобразовать итоговую подпись в base64 | ❌ Выключено |

> THYSS использует plain SHA-256 (не HMAC). Все три галочки должны быть сняты.

---

### 3.7. Раздел «Уведомление»

| Поле | Значение |
|---|---|
| **Признак успешного платежа → Поле** | `status` |
| **Признак успешного платежа → Значение** | `ok` |
| Идентификатор транзакции | *(пусто)* |
| Ответ об успехе уведомления | `OK` *(по умолчанию — не трогать)* |
| Ответ об ошибке уведомления | `ERROR` *(по умолчанию — не трогать)* |

> THYSS возвращает `{"status":"ok", ...}` при успешном создании сессии — именно по этому значению Tilda определяет успех.

---

### 3.8. Настройки формы

| Опция | Значение |
|---|---|
| Отображать поле «Секрет для отправки уведомлений» | ❌ Не включать |
| Отображать поле «URL страницы успеха» | ❌ Не включать |
| Отображать поле «URL отказа» | ❌ Не включать |
| Отображать поле «Тестовый режим» | ✅ Включить на период тестирования, отключить в проде |

> URL после успешной оплаты настраивается в THYSS дашборде → Настройки → Интеграция → Tilda → «URL после оплаты».

---

### 3.9. Заголовок и сохранение

#### Заголовок

**Значение:**

```
THYSS
```

> Этот текст отображается в корзине Tilda как название метода оплаты. Замените дефолтный текст «Новая платёжная система» на `THYSS`.

---

#### Сохранение

Нажмите кнопку **«Добавить»** в нижней части страницы.

## 4. Настройка разрешённых доменов в THYSS

Вернитесь в **THYSS Дашборд → Настройки → Интеграция → вкладка «Tilda»**.

Найдите поле **«Разрешённые домены»** и укажите домен вашего Tilda-сайта.

**Формат:** только домен, без протокола и слеша

```
myshop.ru
```

Если у вас несколько сайтов — каждый домен с новой строки:

```
myshop.ru
shop.myshop.ru
tilda.myshop.ru
```

> ⚠️ Это поле **обязательно**. Без указанного домена все запросы от Tilda будут заблокированы с ошибкой «Invalid origin».
>
> **Частая ошибка:** запись вида `https://myshop.ru` — протокол указывать не нужно.

Нажмите **«Сохранить»**.

## 5. Публикация и тестирование

### 5.1. Публикация сайта

После сохранения всех настроек необходимо опубликовать сайт в Tilda:

1. Откройте редактор Tilda
2. Нажмите кнопку **«Опубликовать»**

> Изменения в настройках применяются только после публикации. Без этого шага новая платёжная система не заработает.

---

### 5.2. Тестовый заказ

1. Откройте опубликованный сайт в браузере
2. Добавьте товар в корзину
3. Заполните форму
4. Нажмите «Оплатить»

**Ожидаемое поведение:**
- Браузер перенаправляется на `pay.thyss.xyz`
- Отображается страница оплаты с суммой в USDT и QR-кодом
- Таймер показывает 30 минут до истечения сессии

---

### 5.3. Проверка в THYSS дашборде

Перейдите в **THYSS → Платежи**.

Должна появиться новая транзакция:
- Статус: `Pending`
- Источник: `tilda`
- Сумма: в USDT

После того как покупатель оплатит — статус изменится на `Success` в течение 1–2 минут.

> **Важно:** статус заказа в Tilda-кабинете останется «Ожидает оплаты» даже после успешной оплаты — это ожидаемое поведение. Tilda не получает обратный callback от THYSS. Актуальный статус всегда в **THYSS дашборде**.

## 7. Часто задаваемые вопросы

**— Ошибка «Invalid signature» при оформлении заказа**

Проверьте по порядку:
1. Алгоритм шифрования — должен быть `SHA-256`, а не MD5
2. «Добавлять секрет в подпись» — должен быть `Последним элементом`, а не `Нет`
3. Чекбокс HMAC — должен быть **выключен**
4. В поле «Исключить поля» указано `secret` (не `signature`)

---

**— Браузер показывает JSON вместо страницы оплаты**

Это происходит при неправильном типе блока в Tilda. Попробуйте использовать стандартный блок корзины (ST100). THYSS автоматически определяет браузерные запросы и возвращает redirect.

---

**— Покупатель заплатил, но статус в Tilda-кабинете «Ожидает»**

Ожидаемое поведение. Tilda не получает callback от THYSS. Смотрите реальный статус в **THYSS → Платежи**.

---

**— Ошибка «Invalid origin» / заблокированный домен**

Проверьте поле «Разрешённые домены» в THYSS. Домен указывается без `https://`. Пример: `myshop.ru`, не `https://myshop.ru`.

---

**— В какой валюте принимается оплата?**

Всегда в **USDT (TON-сеть)**. Если заказ в рублях — THYSS конвертирует по актуальному курсу в момент создания сессии с буфером +1%. Курс фиксируется на 30 минут.

---

**— Можно ли подключить несколько Tilda-сайтов к одному THYSS-аккаунту?**

Да. Один Endpoint URL работает для всех сайтов — просто добавьте все домены в поле «Разрешённые домены». Все платежи будут приходить на один кошелёк.

---

**— Что происходит, если покупатель не заплатил за 30 минут?**

Сессия истекает. Покупатель видит сообщение об истечении срока. Для повторной оплаты нужно оформить заказ заново.

## 8. Техническая справка

> Этот раздел предназначен для разработчиков. При стандартной настройке через Tilda он не нужен.

---

### 8.1. Алгоритм проверки подписи

THYSS проверяет подпись следующим образом:

```
1. Взять все поля POST-запроса, кроме поля "secret"
2. Отсортировать по имени поля в лексикографическом порядке
   (ASCII: заглавные буквы A–Z идут перед строчными a–z)
   Пример: Email, Name, Phone, Receipt, currency, login, orderid, payment
3. Взять только значения полей → склеить без разделителя
4. Добавить Secret Key в конец строки
5. Вычислить SHA-256 → результат в lowercase hex
6. Сравнить с полем "secret" из запроса
```

**Пример вычисления:**

```
Поля (отсортированы лексикографически):
  Email=customer@example.com, Name=John, currency=RUB, orderid=123, payment=1500

Конкатенация значений: "customer@example.comJohnRUB1231500"
Добавляем Secret Key:  "customer@example.comJohnRUB1231500sk_live_xxx..."
SHA-256 → "a3f1c9..."
```

> **Важно:** THYSS использует лексикографическую сортировку (ASCII), а не `localeCompare`. Заглавные буквы (`Email`, `Name`) идут **перед** строчными (`currency`, `login`).

---

### 8.2. Структура POST-запроса от Tilda

**Метод:** `POST`  
**Content-Type:** `application/x-www-form-urlencoded`  
**URL:** `https://api.thyss.xyz/integrations/tilda?pk={ваш_public_key}`

| Поле | Тип | Описание |
|---|---|---|
| `orderid` | string | Номер заказа в Tilda |
| `payment` | string | Сумма заказа |
| `currency` | string | Валюта (`RUB`, `USD`, `USDT`) |
| `secret` | string | SHA-256 подпись |
| `login` | string | Public Key мерчанта |
| `test` | string | `"1"` в тестовом режиме |
| `formid` | string | Идентификатор формы Tilda |
| `Receipt` | string | Квитанция Tilda |
| `Name` | string | Имя покупателя |
| `Email` | string | Email покупателя |
| `Phone` | string | Телефон покупателя |
| `Address` | string | Адрес доставки |
| `City` | string | Город |
| `Zip` / `Postcode` | string | Почтовый индекс |
| `State` / `Region` | string | Регион |
| `Country` | string | Страна |
| `products` | JSON string | Массив товаров корзины |

---

### 8.3. Ответ THYSS

**При успехе (браузерный запрос):**
```
HTTP/1.1 302 Found
Location: https://pay.thyss.xyz?cs=cs_live_...
```

**При успехе (API-запрос):**
```json
{
  "status": "ok",
  "payment": {
    "orderid": "23969993:1234567890",
    "url": "https://pay.thyss.xyz?cs=cs_live_..."
  }
}
```

**При ошибке:**
```json
{
  "status": "error",
  "message": "Invalid signature"
}
```

**Коды возможных ошибок:**

| Сообщение | Причина |
|---|---|
| `Invalid signature` | Неверная подпись — проверьте алгоритм и настройки |
| `Missing payment amount` | Поле `payment` не замаплено |
| `Merchant not found` | Неверный `?pk=` в URL |
| `Monthly transaction limit reached` | Исчерпан лимит тарифа START (10 транзакций/мес) |
| `Merchant wallet not configured` | Не задан TON-кошелёк в THYSS |
